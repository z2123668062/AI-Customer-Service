# 智路由 AI 客服系统：V4.0 架构演进心得

> 本文档不是严格的执行排期，而是一份**架构设计思考记录**。
> 记录了如果要在 V3.0 的基础上再走一步，我会把一个单体 AI 应用拆解、重组、进化为分布式系统的设计取舍与个人见解。

---

## 一、核心追问：单体到底还有没有未来？

V3.0 把单体做到了极致——分层清晰、可测、可观测、安全纵深完善。但做到极致之后，单体的硬边界会浮现出来：

| 维度 | 单体极限 | V4.0 要解决的问题 |
|------|---------|-----------------|
| **扩容** | 只能整体扩，RAG 吃内存、工具调用吃 CPU、推理吃 GPU，无法按需分配 | 独立服务，各自扩缩 |
| **迭代速度** | 改一行工具代码要重新部署整个推理链路 | 每个服务独立发布 |
| **故障隔离** | Redis 挂了→路由炸了→聊天崩了→全站挂 | 局部故障不扩散 |
| **技术异构** | 所有代码必须是 Python，无法用 Go 写高并发网关、Rust 写推理引擎 | 服务间只认协议，不认语言 |

但**拆不是免费的**——网络开销、一致性成本、运维复杂度都会上升。V4.0 的核心命题不是"拆得越细越好"，而是**在合适的地方拆，在不该拆的地方守住**。

## 三、七个核心变化

### 3.1 从"巨石后端"到"网关 + 微服务"

**当前状态**：FastAPI 单体，所有请求直连后端。

**V4.0 做法**：

```
用户 → API Gateway → Orchestrator → Agent A
                                   → Agent B
                                   → Agent C
```

**API Gateway** 用 Go 写（或者用现成的 Kong/APISIX），职责纯粹：
- TLS 终结 + 限流 + 鉴权
- 请求转发 + 响应聚合
- 协议转换（WebSocket ↔ SSE）

**Orchestrator** 是 V4.0 最核心的新服务。它不是一个"路由器"，而是一个**工作流引擎**——它决定：
- 按什么顺序调用哪些 Agent
- Agent 之间如何传递上下文
- 某个 Agent 失败了是重试、降级还是报错
- 什么时候需要多轮 Agent 调用（Agent → 结果 → 再 Agent）

**个人心得**：单体的路由是 if-else，分布式的编排是状态机。前者简单可控，后者灵活但复杂。V4.0 最需要警惕的就是"为了分布式而分布式"——如果只有 2 个 Agent，单体 + 策略模式完全够用。

---

### 3.2 从"单 Agent"到"多 Agent 编排"

**当前状态**：一个 `analyze_intent` 函数做路由，一个 `execute_tool_call` 做工具调用。

**V4.0 做法**：

每个 Agent 是一个**独立部署的微服务**，拥有：
- 独立的 System Prompt（维护在自己的配置仓库中）
- 独立的模型路由策略（简单问题用小模型省钱，复杂问题用大模型）
- 独立的超时和重试策略
- 独立的监控面板

**关键新增——Coordinator Agent（协调者）**：

```
用户: "帮我查一下北京的天气，顺便看看我昨天的订单到哪了"

Coordinator 收到请求:
  1. 拆解为两个子任务: [天气查询, 订单查询]
  2. 并行派发给 Tool Agent
  3. 收集结果
  4. 判断是否需要补充（比如订单没查到，是不是需要追问订单号）
  5. 如果需要追问 → 回复用户等待输入
  6. 如果不需要 → 聚合结果，交给 Polish Agent 包装人话
```

**Claude Code 参考**：Claude Code 的架构中有 Fork Subagent 模式——主 Agent 可以派生子 Agent 并行执行任务，子 Agent 有独立的上下文和文件系统权限。V4.0 的 Coordinator 可以借鉴这个"分治-聚合"模式。

**个人心得**：多 Agent 不是"把一个大 prompt 拆成几个小 prompt"。真正的难点在于**上下文传递**——子 Agent 看到了哪些信息？主 Agent 怎么知道子 Agent 执行完了？子 Agent 的结果冲突了怎么办？这些不是"拆开"就能解决的，需要设计明确的协议。

---

### 3.3 从"单一模型"到"模型网关 + 分级路由"

**当前状态**：全链路只用 `glm-4-flash`。

**V4.0 做法**：

引入 **Model Gateway（模型网关）** 服务，统一管理所有模型调用：

```
                    ┌──────────────┐
                    │  Model Gateway│
                    │  (统一接口)    │
                    └──────┬───────┘
              ┌────────────┼────────────┐
              ▼            ▼            ▼
         glm-4-flash    glm-4-plus   本地模型
         (低成本)       (高质量)      (离线/降级)
```

**分级路由策略**：

| 场景 | 推荐模型 | 原因 |
|------|---------|------|
| 意图路由 | glm-4-flash | 简单分类，足够 |
| 闲聊 | glm-4-flash | 不需要太强 |
| RAG 答案生成 | glm-4-plus（或同级别） | 答案质量直接影响用户体验 |
| 工具调度 | glm-4-flash | function calling 足够 |
| 语言包装 | glm-4-plus | 自然度和语气的重要性被低估了 |
| 会话摘要 | glm-4-flash | 离线任务，不赶时间 |
| 降级/兜底 | 本地 Qwen2.5 | 外部 API 全挂时的最后防线 |

**个人心得**：很多系统高估了"最强模型"的价值，低估了"最合适的模型"的价值。在 V4.0 的架构中，模型不再是"一个配置项"，而是一个**可路由的资源**。这需要 Model Gateway 做好三件事：统一的调用协议、统一的 Token 计量、统一的错误处理（包括模型级别的熔断）。

---

### 3.4 从"请求-响应"到"事件驱动"

**当前状态**：HTTP 请求驱动一切，chat.py 接收请求→处理→返回。

**V4.0 做法**：

引入消息队列（RabbitMQ / Kafka），将核心流程变为异步事件流：

```
用户发消息 → HTTP → Gateway → 发出 "message.received" 事件
                                  │
                                  ▼
                           Orchestrator 消费事件
                                  │
                          ┌───────┴───────┐
                          ▼               ▼
                    "agent.required"    "history.required"
                          │               │
                          ▼               ▼
                      Tool Agent       Memory Service
                          │               │
                          └───────┬───────┘
                                  ▼
                          "response.ready" 事件
                                  │
                                  ▼
                            推送回用户 (SSE)
```

**哪些场景适合事件驱动**：

| 场景 | 当前做法 | 事件驱动做法 | 价值 |
|------|---------|------------|------|
| 转人工通知 | 无 | 发出 `human_handoff.required`，客服系统消费 | 解耦 |
| 长期记忆归档 | BackgroundTasks | 发出 `history.archive.required`，异步消费 | 可靠性提升 |
| 工具调用失败告警 | 日志 | 发出 `tool.failure`，告警服务消费 | 实时通知 |
| 全量历史回溯 | HTTP 直查 MySQL | 发出 `history.query.required`，走 CQRS | 读库不压写库 |

**个人心得**：不要一上来就全部异步化。对于"用户发了消息要立即看到回复"的路径，**同步是正确的方式**——你不需要 Kafka 来让聊天变慢。事件驱动应该从"旁路"开始切入：归档、告警、通知、审计这些**不阻塞主流程**的事情先改。

---

### 3.5 从"单层记忆"到"CQRS 记忆架构"

**当前状态**：Redis 短期 + MySQL 长期，但查询路径单一。

**V4.0 做法**：

读写分离（CQRS - Command Query Responsibility Segregation）：

```
写入路径（Command）：
    对话 → Orchestrator → Memory Service → Redis(热) + MySQL(冷)

读取路径（Query）：
    实时读取：用户当前会话 → 查 Redis（滑动窗口）
    历史读取：用户查昨天对话 → 查 MySQL（全量）
    检索读取：RAG 需要相关历史 → 查向量化记忆（Embedding 后存 Qdrant）
```

**关键新增——记忆得分**：

不是所有记忆都同等重要。每条消息写入时，Memory Service 给它打一个"记忆保留分数"：

```
得分范围 0-1：
  0.9+  → 永久保留在 Redis（用户偏好的、明确要求的）
  0.5-0.9 → 保留在 MySQL，RAG 检索时可见
  0.0-0.5 → 只保留 N 天，到期清理

打分因子：
  - 是否包含明确指令（"记住我住在北京"）→ 高分
  - 是否被后续对话引用（"我刚才说的那个地址"）→ 高分
  - 是否属于日常闲聊（"今天天气真好"）→ 低分
```

**Claude Code 参考**：Claude Code 的记忆策略是**"只记偏好不记代码"**，用 LLM 评分替代启发式规则做召回。它不会记住你某次对话的全部内容，但会记住"用户喜欢简洁的回答"这种元偏好。

**个人心得**：大多数 AI 系统的记忆设计都是"存了就完了"，从来没想过"该忘什么"。V4.0 的记忆架构最核心的设计不是存储引擎的选择，而是**遗忘策略的设计**——一个好的遗忘策略比一个好的存储引擎更能提升用户体验。

---

### 3.6 从"硬编码工具"到"声明式工具市场"

**当前状态**：工具在代码中注册，Schema 和执行逻辑耦合。

**V4.0 做法**：

把工具做成一个**"轻量级工具市场"**：

```
每个工具 = 一个独立的配置文件 + 一个独立的执行单元

tool_configs/
  └── weather/
      ├── manifest.yaml    # 名称、描述、参数 Schema、超时、重试策略
      ├── handler.py       # 执行逻辑
      └── test_cases.yaml  # 测试用例（用于自动化回归）
  └── exchange_rate/
      ├── manifest.yaml
      ├── handler.py
      └── test_cases.yaml
  └── order_query/
      ├── manifest.yaml
      ├── handler.py
      └── test_cases.yaml
```

**运行时**：Tool Agent 启动时扫描 `tool_configs/` 目录，动态加载所有工具。新增一个工具只需要写一个文件夹，不需要改代码。

**工具之间的编排**：有些场景需要组合工具。定义一个简单的 DSL（或者就 YAML）：

```yaml
workflows:
  - name: "售后全链路"
    steps:
      - tool: get_order_status
      - condition: "订单状态 == '已发货'"
        then:
          - tool: get_logistics_info
      - condition: "订单状态 == '退款中'"
        then:
          - tool: get_refund_progress
      - final: polish_response
```

**个人心得**：工具多了之后，最大的成本不是写工具的代码，而是**维护工具的文档和测试**。V4.0 如果只做一件事，我建议先做工具的声明式注册 + 自动生成 API 文档。这和"不做重复劳动"的原则是一致的——工具的 Schema 已经在代码里了，为什么还要手动写文档？

### 3.7 从"手动运维"到"平台工程"

**当前状态**：本地跑、手动重启、日志去文件里翻。

**V4.0 做法**：

```
k8s 部署（或 Docker Compose 升级版）：
  
  FastAPI       2 副本    → 水平扩缩
  Orchestrator  2 副本    → 水平扩缩
  RAG Agent     1 副本    → 资源密集，独立调度
  Memory        1 副本    → 有状态，走 StatefulSet
  Redis                      → 有状态
  MySQL                      → 有状态
  Qdrant                     → 有状态
```

**可观测性升级**：
- **OpenTelemetry**：统一 Tracing 标准，Grafana Tempo 做 trace 存储
- **Prometheus + Grafana**：指标可视化
- **Loki**：日志聚合（替代去服务器翻文件）
- **告警规则**：模型调用错误率 > 5% 触发告警、P95 延迟 > 5s 触发告警

**CI/CD 管线**：
```
Push → 自动化测试 → 构建镜像 → 部署到 Dev 环境 → 
E2E 测试 → 部署到 Staging → 人工确认 → 灰度发布到 Production
```

**个人心得**：DevOps 是 V4.0 中最"吃力但最不讨好"的部分——用户看不到，业务方不关心，但**没有它你不敢改代码**。如果你只有一个人维护这个系统，Docker Compose + 简单的 CI 脚本比 K8s 更实际。K8s 适合团队，不适合个人开发者。

## 四、哪些不该拆？

V4.0 容易犯的错误是"什么都拆"。以下是我个人认为应该**保持单体或轻量拆分**的部分：

### 4.1 安全审校不要拆成独立服务

为什么？安全审校是**横切关注点（Cross-cutting Concern）**，不是业务服务。把它做成独立的 Sidecar 或中间件，而不是一个需要网络调用的微服务。每次对话都要网络调用安全服务，延迟和可靠性都是问题。

### 4.2 语言组织（Polish）不要拆成独立服务

为什么？语言组织的调用频率极高（每次回复都需要），而且它与 Agent 的交互深度绑定——它需要知道 Agent 的完整上下文才能写出好的人话。拆出去只会引入不必要的网络开销。

**一个更好的选择**：把 Polish 做成 Agent 内部的插件/策略，而不是一个独立的网络服务。V3.0 方案中把它独立为一个模块（同一个进程内调用）是正确的方式。

### 4.3 路由分发不要拆成独立服务

为什么？路由是整个系统的**决策入口**，把它拆成独立服务意味着每次请求都多一次网络调用。更重要的是，路由需要访问完整的请求上下文（Token 信息、历史记忆、用户偏好），这些数据在网络传输中要么冗余（全传），要么丢失（不全传）。

**一个更好的选择**：路由作为 Orchestrator 的内置能力。Orchestrator 本身就是一个独立服务，路由是它的一个子模块——逻辑隔离但进程不隔离。

## 五、V4.0 的技术债（做了比不做好，但不做也能活）

这里列出的不是必须项，而是**"有钱有闲"时可以琢磨的方向**：

| 方向 | 价值 | 成本 | 建议 |
|------|------|------|------|
| gRPC 替代 HTTP 做服务间通信 | 性能提升 30-50% | 需要维护 proto 文件 | 初期 HTTP 够用 |
| WebSocket 替代 SSE | 双向通信，可做"服务器主动推送" | 连接管理复杂 | SSE 对客服场景已经够好 |
| GraphQL 聚合层 | 前端按需取字段，减少带宽 | 学习曲线 | 如果只有 Streamlit，不需要 |
| 本地 GPU 推理 | 零外部依赖，无 API 成本 | 需要硬件 | 有 GPU 可以试试，没有就算了 |
| 混沌工程 | 验证系统在极端条件下的表现 | 高 | V4.0 阶段太早了 |

## 六、结语：V3.0 到 V4.0 的核心转变

如果要用一句话总结 V4.0 和之前版本的本质区别：

- **V1.0** 解决"有没有"——能不能跑通一条链路
- **V2.0** 解决"稳不稳"——能不能上线用
- **V3.0** 解决"好不好"——能不能维护、能不能信任
- **V4.0** 解决"大不大"——能不能多人协作、能不能承载团队

V4.0 不再是一个"个人项目"的架构，而是一个"团队项目"的架构。它引入的复杂度——服务拆分、事件驱动、多 Agent 编排——在个人项目中可能是过度设计，但在一个 5-10 人的团队中，这些是成长的必经之路。

如果你正在面试，V3.0 的深度 > V4.0 的广度。面试官更想听你把一条链路讲透，而不是把十条链路都点到为止。V4.0 的视角更适合你**入职之后**，当你在真实团队中遇到"一个 20 万行的 FastAPI 文件该不该拆"这个问题时，翻出来看看。
