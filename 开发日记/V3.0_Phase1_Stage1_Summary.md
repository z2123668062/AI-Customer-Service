# V3.0 第一期纯干货日记：并发模型修复与安全纵深加固

## 核心目标

V3.0 的执行原则是"填坑优先"——修补之前留下的缺陷永远比引入新模块重要。第一期完成了填坑阶段的两个 P0 任务：并发模型修复（任务 1.1）和安全漏洞修复（任务 1.2），以及在全链路排查中发现并修复了 Redis 客户端同步阻塞的问题。

共改动 14 个文件，消除了两类系统性隐患：**事件循环被同步调用阻塞**和**密钥硬编码与鉴权缺失**。

---

## 核心变革一：并发模型全线异步化

### 问题：同步调用混在异步事件循环里

系统运行在 Uvicorn 单 worker 单事件循环上，所有用户共享同一个事件循环。如果某个请求里混入了同步阻塞调用，整个事件循环会被卡住，其他用户排队等待。

之前代码中有三处明显的阻塞源头：

1. `tool_service.py` 用同步 `OpenAI()` 客户端调用大模型，在 `async def execute_tool_call` 里直接同步调用，没有 `await`
2. `rag_service.py` 的 `query_knowledge` 是普通 `def`，在异步 `stream_generator` 里直接调用，LlamaIndex 的 `query_engine.query()` 是 CPU 密集 + 同步操作
3. `chat.py` 的所有外部调用（大模型、RAG、工具）都没有超时熔断，任何一个卡死了整个 `stream_generator` 就挂在那

### 修复方式

**tool_service.py**：`OpenAI` 换成 `AsyncOpenAI`，两个 `client.chat.completions.create()` 加 `await`。这个改动最小，只换了导入语句和两个 await。

**rag_service.py**：`query_knowledge` 从 `def` 改成 `async def`，内部同步的 `query_engine.query()` 用 `asyncio.to_thread()` 丢到线程池执行。这里不能用简单的 `await`——因为 `query_engine.query()` 本身不是协程，`await` 一个普通函数会直接报 `TypeError`。`asyncio.to_thread` 是 Python 3.9 引入的标准库函数，作用是把同步函数扔给线程池，返回一个协程让你 await，主事件循环不会被卡住。

**chat.py**：`stream_generator` 里三个分支（chitchat、kb_qa、tool）全部用 `asyncio.wait_for()` 包上超时——大模型调用 15 秒、RAG 查询 10 秒、工具调用 15 秒。超时后降级返回友好提示（"抱歉，我思考得有点久了""抱歉，知识库查询超时了"），系统不崩溃。另外 kb_qa 分支原来 `query_knowledge` 前面漏了 `await`，这次也一并补上了。

### 顺便揪出的更深层问题：Redis 全链路同步

改完上面三处后，继续排查事件循环上的其他调用，发现**所有 Redis 操作都是漏网之鱼**——`memory.py` 用的是同步 `redis.Redis()`，`rpush`、`lrange`、`incr`、`hset`、`lock.acquire` 全都是同步阻塞调用。虽然 Redis 通常只有 1-5ms，但网络抖动时一样能卡住事件循环。

涉及 5 个文件，改法统一：

**memory.py（底层）**：`redis.Redis` 换成 `redis.asyncio.Redis`，所有 4 个函数（`add_message`、`get_history`、`get_history_count`、`undo_last_turn`）改为 `async def`，5 个 Redis 操作加 `await`

**ratelimit_service.py**：`check_chat_limit` 和 `check_session_limit` 改为 `async def`

**session_service.py**：`create_session`、`list_sessions`、`update_session_title`、`delete_session` 全部改为 `async def`

**chat.py（第二次改）**：`lock.acquire()`、`lock.release()`、`check_chat_limit()`、`add_message()`、`get_history()`、`get_history_count()` 全部加 `await`

**sessions.py**：`session_service` 的 4 个调用和 `check_session_limit` 加 `await`

改完后遇到 `redis==7.4.0` 的类型标注 bug——`redis.asyncio` 的 type stub 把一些异步方法标成了同步，导致 IDE 报 `"int" is not awaitable` 的误报。升级到 `redis==8.0.0` 后解决。

### 改造前后对比

改之前：同步调用阻塞事件循环，多用户并发时串行排队。用户 A 调大模型时，用户 B 无法被处理。

改之后：事件循环上不存在任何同步的网络/IO 调用。`AsyncOpenAI` 发请求时事件循环去处理其他用户，`asyncio.to_thread` 把 CPU 密集操作扔到其他线程，`asyncio.wait_for` 确保任何卡死的情况都不会超过超时阈值。

## 核心变革二：安全漏洞修复

### 问题：三处安全隐患

代码中存在三类安全问题——密钥耦合、硬编码、鉴权缺失。

**密钥耦合**：`auth_service.py` 的 `JWT_SECRET_KEY = settings.ZHIPU_API_KEY[::-1]`，JWT 签名密钥等于智谱 API Key 的反转。API Key 泄露则 Token 可伪造，两个凭据绑死了。

**硬编码**：`tool_service.py` 第 15 行直接写死了高德天气 Key（`GAODE_WEATHER_KEY = "dce5bfc8..."`），提交到 Git 所有人都能看到。

**鉴权缺失**：`POST /api/v1/kb/build` 没有任何认证，谁都能触发知识库重建。`main.py` 没有 CORS 配置，前端跨域时会报错。`/register` 和 `/login` 没有防暴力破解。

### 修复方式

**config.py**：新增三个配置字段——`JWT_SECRET_KEY`（独立随机字符串）、`GAODE_WEATHER_KEY`（高德 Key，可选）、`ADMIN_TOKEN`（管理接口令牌）。所有敏感信息统一走 `.env`，代码中不留任何真实密钥。

**auth_service.py**：`JWT_SECRET_KEY` 从 `settings.ZHIPU_API_KEY[::-1]` 改为 `settings.JWT_SECRET_KEY`。JWT 和 API Key 彻底解耦。

**tool_service.py**：移除硬编码字符串，改为 `GAODE_WEATHER_KEY = settings.GAODE_WEATHER_KEY`。如果 `.env` 里没配这个字段，`settings.GAODE_WEATHER_KEY` 就是空字符串（因为定义了默认值 `""`），工具调用时 API 请求自然会失败，不会暴露密钥。

**kb.py**：`POST /api/v1/kb/build` 增加 `X-Admin-Token` 请求头校验，与 `settings.ADMIN_TOKEN` 比对，不匹配返回 403。调用方（管理员）手动在请求头中携带这个令牌。

**main.py**：增加 `CORSMiddleware`，当前全放行（`allow_origins=["*"]`），生产环境可以收紧为具体域名列表。

**auth.py**：`/register` 和 `/login` 增加 IP 级别限流，每 IP 每分钟最多 5 次。复用 `redis_client.incr` + `expire` 的固定窗口计数器模式，与 `ratelimit_service.py` 的技术方案一致。

## 踩坑实录

### redis.asyncio 类型标注误报

`redis==7.4.0` 的 `redis.asyncio` 模块类型标注文件（`.pyi` 文件）写得不准确——部分异步方法被标注为同步函数的返回类型。升级到 `redis==8.0.0` 后，类型标注全部修复，IDE 不再报 `"int" is not awaitable`。这个错误只在 IDE 的类型检查中体现，运行时没有任何问题。

### asyncio.to_thread 的使用场景

`rag_service.py` 的 `query_knowledge` 改造时，一开始直接写了 `response = await query_engine.query(question)`，但 LlamaIndex 的 `query_engine.query()` 是同步方法，不是协程。对同步函数用 `await` 导致 `TypeError`。修复方案是用 `asyncio.to_thread(query_engine.query, question)`——将同步函数放到线程池执行，返回一个可 await 的协程对象。

### 直接 await 同步函数的误区

做 Redis 异步化时有一个认知需要澄清：不是把某个导入换成异步版就完事了，上游所有调用点都必须加 `await`。这次改动的 5 个文件中，`chat.py` 和 `sessions.py` 的改动就是典型的"连坐效应"——即使底层函数改了 `async def`，如果调用点没加 `await`，函数只会返回一个协程对象而不是实际结果，且不会报错（因为不 await 协程也算合法的 Python 语法）。这意味着一不小心就会漏掉。

## 架构反思

### 1. 异步改造的传染性

异步有一个"毒性传播"的特性——底层一个函数改成 `async def`，中间层调用必须加 `await`，上层调用也必须加 `await`，一路传到 API 端点。这次从 `memory.py` 改到 `chat.py` 跨了 5 个文件就是典型的例子。

这次能一次性改完是因为我们已经知道了完整的调用链。如果以后新加一个服务层函数，一开始就写成 `async def` 会省去很多"补 await"的工作——反过来从同步改异步的改造成本远高于一开始就写成异步。

### 2. 安全是配置问题，不是代码问题

这次安全加固的核心思路是：**密钥和凭据不应该出现在代码中。** 代码是版本化的、共享的、可回溯的，而 `.env` 是本机的、不提交的、按环境区分的。把密钥从代码移到配置，不仅仅是隐藏了字符串，更是让"不同环境用不同密钥"成为可能——开发环境、测试环境、生产环境各有各的 `.env`，代码完全不变。

### 3. CORS 的定位

CORS 不是安全防线，是浏览器的同源策略保护机制。它保护的是"用户在浏览器里访问恶意网站时，恶意脚本不能随意读取其他网站的响应"。真正的后端安全靠的是 JWT 鉴权、限流、密钥管理——这些已经在本次任务中做了。CORS 在这里只是确保前后端不同端口/域名下不会莫名其妙报错，配上 `["*"]` 在开发阶段足够，上线前收紧。

### 4. 固定窗口限流的取舍

固定窗口限流（`incr` + `expire`）有一个经典的边界问题：用户在窗口末尾和窗口开头连续发送请求，可能两秒内发出 2 倍限流量的请求。但实现极简单——一行 `incr` 一行 `expire` 搞定。更严格的滑动窗口方案（用 Redis ZSET 记录精确时间戳）实现复杂度高得多且很少在真实的业务场景中产生质的差异。对于每分钟 5-20 次的限制级别，固定窗口的"漏放率"在实际使用中完全可以接受。
