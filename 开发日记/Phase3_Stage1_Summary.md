# 第三期纯干货日记（上）：RAG 检索引擎“冷热分离”与单例防御机制
## 核心目标
解决每次问答都在重新读取本地文件构建特征向量的“性能灾难”，将知识库的构建与检索功能彻底拆分。同时，拔除隐藏在底层的严重资源泄露（文件句柄）隐患，确保系统的响应延迟从“分钟级”降到“毫秒级”。
## 核心思考与架构认知：为什么 1.0 的写法是一场“灾难”？
在我们的初代 MVP 架构中，系统通过一个名为 build_or_load_index 的庞然大物来处理知识库。当用户输入问题触发 kb_qa 路由时，系统会硬生生地：
1. 从硬盘扫描全量文本（如 company_policy.txt）。
2. 让本地大模型把文本切块、逐字进行向量化（Embedding）运算。
3. 把算好的数字阵列强行覆盖式存入 ChromaDB，这才开始查询。
**这场灾难意味着什么？**
就相当于顾客来问招牌菜是啥，厨子每次都要跑到后厨先读完一整本500页的菜谱再回答。不仅拖慢了原本毫秒级的检索速度，如果有100个用户同时发问，极度耗时的并发矩阵运算会让服务器内存瞬间爆炸，导致严重的死锁（如 SQLite 的 Database is locked）。
这让我们深刻领悟了系统拆分的一个金科玉律：**在关键业务流（用户提问链路）中，永远禁止放置重型计算任务**。我们要将“做饭（建库）”与“吃饭（查库）”进行**物理级的冷热隔离**。
---
## 防御推演与实战重构：如何优雅地进行动刀？
系统解耦的核心，是将原有的业务拆成不同频率、不同权限的使用场景。
### 1. 热任务：在线查询的彻底瘦身
对于面向海量并发用户的接口（chat.py 里的问答），它的底线是不做任何构建计算。
我们重写了 rag_service.py 的底层函数 get_readonly_index，**强行禁用了原本的 from_documents()（重头建库指令），全面替代为 from_vector_store()**。这一招让查询动作直接穿透 LlamaIndex 库，直达已构建成型的 Chroma 向量索引。
**效果**：不再出现 Loading weights 的假死卡顿，真正实现了毫秒级的向量搜寻。
### 2. 冷任务：管理员专用的专属建库通道
“做饭”必须移出前台！我们在全新的 kb.py 中写了专门的 @router.post("/build") 接口。
当公司上传了最新的《差旅报销制度》时，管理员主动调用此接口去触发那极度耗时但不可避免的三分钟离线重算流程（build_knowledge_base）。这一操作默默躲在后台，丝毫不影响其它千万人的在线发问。
---
## “破坏与破拆”：全局单例的防御战争
在实现冷热分离后，面试官/导师角色的刁钻提问让我意识到了更恐怖的暗坑。我们在对系统进行无情施压时发现了致命的**文件句柄泄露灾难**：
### 痛点：没关水龙头的“对象生成”
原来我们写了一个每次被调用都会执行 db_client = chromadb.PersistentClient(path=db_dir) 的辅助连库函数。不论是查询还是建库，只要请求一进来，操作系统就会乖乖地打开一条底层物理管道连向本地硬盘。由于没有主动关闭，高并发状态下极容易耗光默认的 65535 个系统文件句柄，引发瘫痪甚至断电级的宕机！
### 破局方法：饿汉式“全局单例模式”    
我们将与硬盘地址绑定和库连接的代码从局部域里生硬拔出，**以全局变量的形式暴露在文件最顶端（声明为静态单例）**。一旦这套 Fastapi 系统在启动的那一微秒挂载完毕，全盘运行期间共享同一个唯一打通的管道（Client），再也没有多余的额外消耗。

## 架构的前瞻反思：我做的重构无懈可击吗？
作为一个有野心的系统，这种看似优雅的应对其实依然面临苛刻的瓶颈。这也是未来走向真实微服务我们要淘汰它们的原因：
*   **饿汉式单例的诅咒**：虽然挂载本地 Chroma 的单例是在瞬间生成的，但如果这一招原封不用搬到依赖外网握手组件（如必须用 async/await 建连的 Async MySQL），它就会在 FastAPI 的生命循环完全拉起之前强行启动，导致报死出 RuntimeError: There is no current event loop 的恶劣闪退。解法必须妥协于更加复杂的**依赖注入（Dependency Injection, DI）和事件懒汉式（Lazy）初始机制**。
*   **不友好的平滑热更**：此方案对动态数据库迁移不友好。如果我们想在不停机的情况下一键把挂载地址切换到另一块硬盘上，静态单例就像死死的枷锁无法更换，也因此引入更成熟的网络存储中间件势在必行！
至此，第三期的底层基石已牢牢打定，RAG 从脚本玩具终于拥有了不惧并发读写的初步抵抗力！接下来，重排（Rerank）与更加工业化的外建向量库在向我们招手。

## 第三期纯干货日记（下）：交叉重排（Rerank）精准容错与 Qdrant 容器化升维

在完成了基础的读写分离后，我们在业务的准确度和架构的扩展性上发起了最终冲锋。

### 1. 业务痛点：大模型“近视眼”与幻觉之源
原来的 Embedding（双塔模型）检索极速但粗糙，它纯粹靠算空间距离找资料。这就导致当搜“北京到上海的高铁”时，它极其容易把“上海到北京的高铁”给高分捞出来，这俩词频和字面太像了！大模型拿到这堆伪证，立马开始一本正经地胡说八道。

**工业级解法：引入交叉重排（Rerank）模型**
我们在 Chroma/Qdrant（一审）与 大模型（结案）中间，强行插入一个“批卷老师”。
让底层的数据库放宽标准，先粗暴捞出 10 条擦边文本。然后让 Reranker 把“用户提问”与“候选文本”死死捆在一句话里送进神经网络（Cross-Encoder，注意力机制全开）。模型会逐字分析两者的逻辑抵触点，打出精准分数，最后只把 Top 2 的王者送给大模型。

### 2. 生态灾难与原生手搓：打破依赖地狱 
在挂载 Reranker 时，我们遭遇了经典的“生态库版本车祸”。官方的 `llama-index-postprocessor-flag-embedding-reranker` 包深度死咬了老版 `transformers` 中被删掉的函数 `prepare_for_model`，导致无论是退回老版本（陷入无 C++ 环境的源码编译死循环），还是更新新版本都疯狂卡死。

**“不求施舍，手搓反杀”**：
与其被套壳三方库裹挟，咱们直接掀桌子，利用底层的真神 `sentence-transformers` 亲手实现了一套 `MyCustomBGEReranker`（继承自 `BaseNodePostprocessor`）。
核心实现极其优雅：拦截捞出来的 `nodes`，把 `query_str` 和 `node_content` 拼成一对，调用 `_model.predict(sentence_pairs)` 计分并覆写排序，最后切片。短短 20 行代码直接免疫一切包装冲突，直取底层核心机制！

### 3. 基建升维：拆除本地 SQLite，上线 Qdrant 微服务 
原先 ChromaDB 会在本地生成 `.sqlite3`，这玩意儿在单机玩玩可以，一旦业务壮大、上了多机并发，多节点同时写一个文件绝对是 `Database Locked` 的死局。
我们利用 `docker-compose.yml` 引入了由 Rust 编写的高性能极简数据库 **Qdrant**，解决 Docker 源被墙后将其放入独立容器，并暴露出 6333 端口。代码里抛弃所有的 `os.path` 本地路径，换上了硬核的网络大动脉：
`GLOBAL_QDRANT_CLIENT = qdrant_client.QdrantClient(host="localhost", port=6333)`
从这一刻起，我们的 RAG 后端不再是一个自嗨的单体脚本，而是真正切入了“异度空间节点通信”的云原生微服务时代。

### 4. 架构师防守拷问：高并发引擎下的深水炸弹
在这套华丽架构下，其实还埋藏着极深的隐患，是应对大厂高压面试的绝佳反思：
*   **算力雪崩与 GIL 锁死**：Rerank 是纯 CPU/GPU 密集型算力巨兽。如果千人并发，受限于 Python 粗糙的 GIL（全局解释器锁），主 Web 节点将被死死卡在矩阵运算上无法处理其他 HTTP 请求。**解决之道**必须是将 Embedding 和 Rerank 剥离成独立的高并发推理微服务（如 Triton Inference Server），系统主服务只做轻量网络请求。
*   **微服务断链与重试防线**：微服务是一把双刃剑。那个高高挂在全局的长连接 `QdrantClient` 若遇上服务器底层的容器一秒闪退，这根旧网线就彻底成了“僵尸”。千万并发涌来，将引发史无前例的 `ConnectionRefusedError` 级联雪崩。因此，为这类网络长连接配置带指数退避和保活心跳的**重试中间件连接池（Retry/Pool）**才是最终的工业级终态。
*   **无状态容器的致命幽灵**：我们在跑 docker-compose 时特意加了 `volumes` 挂载目录！如果缺少这行，只要 Docker 容器一重启，Qdrant 内存里千万条打好的向量数据将直接蒸发！微服务的容器必须被当做随时可能死掉的“无状态”肉鸡，只有持久化存储配置到底层磁盘才是最终的保障。
