# 第二期纯干货日记（上）：记忆系统的重构与短时记忆（Redis）

## 核心目标
彻底拔除项目1.0时代的“毒药”全局变量字典，引入真正的外部高性能内存数据库（Redis），并实现基于滑动窗口的大模型 Token 保护机制。让我们的后端服务从“带状态的单点服务”升级为“无状态且支持水平扩展”的弹性服务。

## 核心思考与认知拔高

### 1. 为什么原有的 `_sessions = {}` 全局字典是“毒药”？
* **内存爆炸（OOM风险）**：全局变量存放于 Python 进程的内存中，一旦用户激增，几万人的聊天记录会迅速把服务器内存撑爆。
* **数据脆弱性**：只要服务器一断电、重启或者更新代码，所有用户的会话将瞬间清零。
* **分布式死穴**：在真实服务器上，为了抗并发，我们会开了多个 Python 进程乃至多台机器跑集群。用户第一句话发到机器A，第二句话被分发到机器B时，机器B的系统内存里根本没有这个人之前的数据。

### 2. 滑动窗口与商业级大模型记忆控制
* **Token限缩与计费保护**：大模型拥有严格的上下文长度限制并且按 Token 收费。通过滑动窗口，我们强行剥离老旧对话，始终只将最近的 `N` 条（我们设定为10条/5轮）送给大模型，防止无意义的记账消耗。
* **商业方案拓展（防失忆）**：针对滑动窗口带来的“历史遗忘”问题，商业闭环通常会在后台开一个廉价大模型，将长历史进行总结（Summarization），然后将用户的偏好提炼提取并永久挂载在 `System Prompt` 之中。

### 3. 高级架构思维：不要拉回本地计算
在限制列表大小时，我们抛弃了“先全量读取到 Python 列表 -> 截取后10个元素 -> 再覆盖写回”的笨重传统做法，而是采取了 **让数据库自己算（Offloading compute to DB）** 的防御性思维。网络 I/O 是最宝贵的，把指令直接交到底层用 C 语言写的 Redis 手中，能达到最极致的性能。

## 核心技术实现：Redis 的 Python 实战

在 Python 与 Redis 的交互中，万物皆不能直接存对象。存入前必须经历 **序列化**（`message.model_dump_json()`），取出后再进行 **反序列化**（`json.loads()` + `ChatMessage(**data)`）。

本期我们吃透了以下几个核心的 Redis List（列表）实战命令：

1. **追加指令：`rpush (Right Push)`**
   * **作用**：将用户的最新对话推送到特定 `session_id` 键对应的列表最右侧（尾部）。
   * **场景**：用户发来新消息时、AI返回新消息时。
2. **修剪指令：`ltrim (List Trim)`**
   * **作用**：对列表进行范围裁切。`ltrim(key, -10, -1)` 代表只保留此列表倒数第 10 个到最后一个元素。多余的旧数据会被 Redis 在底层直接抹去。
   * **场景**：每次 `rpush` 后立刻执行，这就是不用 Python 写逻辑就能实现滑动限长窗口的魔法。
3. **读取指令：`lrange (List Range)`**
   * **作用**：读取指定范围的列表切片。`lrange(key, 0, -1)` 代表无视多长，全量拿到当前列表幸存的所有元素。
   * **场景**：正式向大模型发请求前，从 Redis 中提取组装上下文 `messages`。
4. **长度指令：`llen (List Length)`**
   * **作用**：极速获取列表当前长度，O(1)复杂度。
   * **场景**：统计会话有多少条历史，完全不需要拉取具体数据。
5. **弹出指令：`rpop (Right Pop)`**
   * **作用**：移除并返回列表最右侧（最新）的那一个元素。
   * **场景**：我们手写的“撤回上一轮对话”功能，结合 `llen` 做了边界拦截后，连续使用两次 `rpop` 完美清除了刚刚写反悔的一问一答。

## 第二期纯干货日记（下）：长效记忆的持久化落盘（MySQL + SQLAlchemy）

### 核心认知与架构设计

#### 1. 引入 ORM 替代原生 SQL（SQLAlchemy）
* **实质**：将关系型数据库“对象化”（Object-Relational Mapping）。我们没有打开 Navicat 敲击建表指令，而是通过继承 `DeclarativeBase` 创建了 `ChatRecord` 实体类。
* **价值**：
  * **代码即结构**：在业务逻辑层就可以直观看到表结构，做到自动建表。
  * **解耦数据库**：屏蔽了底层数据库方言差异。今天用 MySQL，明天换 PostgreSQL，只需换一条 `DATABASE_URL`，业务代码零修改。
  * **安全防线**：完全规避了传统拼接 SQL 字符串带来的“SQL注入”安全风险。

#### 2. 全异步数据库兵器（aiomysql + AsyncSession）
* **实质**：打破传统数据库“发请求必须死等结果”的阻塞（Blocking）模型。
* **价值**：通过 `create_async_engine` 和 `AsyncSessionLocal` 搭建的流水线，使得 Python 可以在执行 `await db.commit()` 时，立刻把计算资源让出给其他并发的 Web 请求。这种非阻塞机制是 Python 在单机实现几万 QPS 吞吐量的核心保障。

#### 3. “旁路双写”与冷热分离（BackgroundTasks）
* **痛点**：数据库的磁盘落盘速度极慢如果等 MySQL 写完再给前端回信，接口延迟会剧增。前端立刻请求记录时，如果 MySQL 还没写入，就会发生“数据最终一致性（Eventual Consistency）”错乱。
* **破局方案（甩手掌柜模式）**：
  * **热库防线（Redis）**：高速缓存极速插入并立即生效，前端请求永远查热库，保证毫秒级无缝体验。
  * **冷库归档（MySQL）**：利用 FastAPI 的强悍特性 `BackgroundTasks`，把落盘函数 `save_record_to_db` 扔进后台队列默默存入。
  * **最终效果**：用户得到了急速对话体验，同时我们在数据库留存了不可篡改的永久追溯凭证。

#### 4. 高级除错：分支拦截的“漏水防线”
* **事故复盘**：在 `complaint`（投诉）等特殊的截断分支中，使用了 `return` 提前结束接口。这导致系统只记下了用户的抱怨，却漏记了客服的安抚和补偿。
* **避坑法则**：带有“全局环绕动作”（如写入上下文、双写数据库）的接口，一旦需要中途 `return` 强行跳出，必须在跳出前补全所有的善后（写 Redis 短期记忆、抛出后台双写任务）。这是防范系统状态分裂、商业数据断层的生命线。

## 第二期纯干货日记（进阶篇）：高并发防御与极致兜底

在完成极速双写闭环后，我们通过“破坏性实验”对系统进行了高并发和代码异常的施压，进一步获得了以下商业级防御经验：

### 1. 突破并发盲区：“防止影分身”的 Redis 分布式锁
* **隐蔽的灾难（读写竞态污染）**：由于大模型请求被 `await` 挂起，如果用户在1毫秒内狂点3次发送，3个并发请求会同时读到相同的会话历史前文，然后去请求模型并写回。这会彻底搞乱对话历史的时间轨道（例如连续三句一模一样的提问，连着三句回答）。
* **实战防具**：在进入 API 核心逻辑前，引入 **Redis 分布式锁**。
  * `lock.acquire(blocking=False)`：拿到锁才能进门。没拿到锁（锁孔被上一次请求占着），说明这是恶意的或手抖的短时间连发，直接抛出 `409 HTTPException` 拦截并拒绝。
  * `timeout=10`（生命期防死锁）：设置了10秒的自动销毁时间。防止服务器突然断电导致某个用户的锁永远不解开，被永远关在小黑屋里。

### 2. 终极的生命线防御：`try...finally`
* **痛点**：我们的路由判断可能抛出 500 崩溃异常，特定的拦截可能提前 `return`。如果不在走之前把门（锁）打开，下一个正常的请求也会被挡住。光靠 `except` 是无法兜住所有的意外死法的。
* **破局底层原理**：回归 Python 最绝对的铁律——`finally`。通过将接口业务全部包裹在 `try` 块中，并在底部保留 `finally: lock.release()`，我们做到了一件伟大的事：**无论你的代码是因为什么报错而崩溃，也无论你是从哪个分支安全跳出的，在离开这片战场的最后一刻，都必定会执行锁释放。** 这是任何大型系统资源管理（文件句柄、数据库连接、并发锁）最标准的“强迫症”防御。

### 3. 未来的技术债与思考：背景任务的脆弱性与消息队列（MQ）
* **深度思考（破拆实验带来的反思）**：虽然我们基于 FastAPI 的 `BackgroundTasks` 实现了极速的响应（把落盘动作推迟到后台），但这也引发了一个致命疑问：如果这条消息尚未存入 MySQL，而服务器因为停机或内存耗尽（OOM）瞬间崩溃了怎么办？
* **现状痛点**：在现有的内存后台任务模式下，队列里的数据会随着进程的死亡瞬间蒸发，导致数据库永远丢失了这一条用户的对话记录，形成“状态极其脆弱”的隐患。系统明明已经向前端返回了成功响应，后端数据却没留存。
* **终极解法预言**：我们讨论明确了，当项目走到高阶或需要保证“金融级绝不丢数据”的严苛场景时，必须抛弃内存级 BackgroundTasks，转而**引入专业的消息队列（Message Queue, 如 RabbitMQ 或 Kafka）**。核心机制如下：
  1. 接口接收消息后，先将其持久化打入 MQ。如果投递成功，再给前端返回响应。
  2. 后边有专门独立运行的 Worker 进程去 MQ 里消费并保存到 MySQL。
  3. 即便 Worker 崩溃断电，由于 MQ 没收到 ACK 回执（确认消费），下一次重启时这条消息依然会被重新吐出，真正实现 **“断电也不丢，使命必达”**。
  （注：鉴于现阶段重点在于梳理架构，消息队列的重构门槛较高，暂定为日后的高阶技术演进路线。）
