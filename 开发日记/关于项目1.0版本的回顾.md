# AI_Agent 项目代码架构与执行流程解析

本文档成体系地梳理了本项目目前所包含的所有 Python (`.py`) 文件的职责。为了让您更深刻地理解整个项目，我们将 **按照用户的请求流转与应用初始化顺序** 来进行逐级剖析。

---

## 阶段一：系统基座初始化（项目加载时最先被解析引用的模块）

程序启动或模块被引入时，最底层的数据结构与配置会被先行加载，为后续业务提供基础支持。

### 1. `app/core/config.py`
- **主要作用**：系统统一配置中心。负责在程序启动时，通过解析 `.env` 环境变量文件加载密钥等敏感参数并保存在内存中如果缺少致命项会阻断程序启动（防报错抛锚）。
- **主要引入包**：`os`, `dotenv.load_dotenv`

### 2. `app/models/schemas.py`
- **主要作用**：数据契约模型层（DTO层）。制定前端界面和后端服务器间通信的固定数据格式（Request、Response）以及系统内部状态参数限定。
- **主要引入包**：`pydantic (BaseModel, Field)`, `typing`, `datetime`
- **包含的具体类定义**：
  - `HealthResponse`: 定义了健康检查的心跳响应数据类型。
  - `ChatRequest` / `ChatResponse`: 对话接口出入的标准结构（请求体需要携带 session_id 以及 message，并封装会话历史数、追踪ID返回）。
  - `ChatMessage`: 标准化对话角色的模型记录格式。
  - `RouterResult`: 锁定 LLM 路由器的返回格式，严格要求 "intent"（工具/知识库/闲聊/投诉四选一）和 "keywords"（关键字）字典结构。

### 3. `app/services/safety_service.py`
- **主要作用**：安全控制底座。模块加载时会预先在内存里编译“敏感词字典树（TrieTree）”。拦截诸如 “作弊”、"违禁词" 等恶意词汇。
- **主要引入包**：`logging`, `re` 
- **包含的函数/类**：
  - `SensitiveWordTrie`, `TrieNode` 类: 用于实现字典查找时间复杂度仅 $O(N)$ 的高效敏感词快速匹配算法。
  - `normalize_text(text)`: 文本清洗拦截（去除符号空格，防绕过）。
  - `check_input_safety(text)`: 执行用户输入的拦截检查。
  - `get_safety_rejection_message()`: 提取固定的违规拦截统一文本。

### 4. `app/core/memory.py`
- **主要作用**：轻量级内存会话隔离。由于目前第一期项目未接入复杂 DB，利用 Python 的原存 Dict 将各 UUID Session 层切分。
- **主要引��包**：`typing`, `app.models.schemas.ChatMessage`
- **包含的函数**：
  - `add_message(...)`: 格式化并向用户的专属内存记录区追加一条历史记录。
  - `get_history(...)`: 返回选定客户端的对话记录数组。
  - `get_history_count(...)`: 统计该会话有几条来回的历史消息。

## 阶段二：服务器入口启动控制层

基石建设完毕以后，启动真正的外部可连接程序入口。

### 1. `app/main.py`
- **主要作用**：FastAPI 后端核心引导入口。通过挂载各个独立路由将系统整合，同时接管三层“全局异常”（如业务 HTTP 状态码错误 / Pydantic 字段验证请求错误 / 万能通用 Exception）进行友好式 JSON 返回，建立底线。
- **主要引入包**：`fastapi`, `logging`, `app.api.v1.endpoints.chat`
- **包含的函数**：
  - `health_check()`: `GET /health` 探针检查程序死活。
  - `validation_exception_handler` / `http_exception_handler` / `global_exception_handler`: FastAPI 的全局错误拦截钩子。

### 2. `app/api/v1/endpoints/chat.py`
- **主要作用**：用户提问到达之后的“核心分拣 Controller”。所有发送至 `/api/v1/chat` 的请求都首先进入该函数，按照“安全审校 -> 大模型意图路由分析 -> 对应业���分支流转业务”的完整生命流走一遍。
- **主要引入包**：当前项目中定义的大量 Schema 与全部 Service 模块。
- **核心入口函数**：
  - `chat_endpoint(request: ChatRequest)`: 收取入参并生成业务 `trace_id` -> 调用 `safety_service` 安全放通检查 -> 调用 `memory` 记录来信 -> 呼叫并等待 `router_service` 返回意图意向字典 -> 根据字典分支走向对应的 Service (下面四个模块之一) -> 调用 `memory` 记录 AI 产出回复 -> 封装最终的结果体给前端。

## 阶段三：具体意图分支处理层 (Service层)

当 `chat.py` 解析到分流信号时，分别流转至下面三大工作模块之一。

### 1. `app/services/router_service.py`
- **主要作用**：智能意图分析与闲聊发生器引擎。负责充当项目架构中“判断用户问题类型”的大脑，向智谱大模型抛送 JSON 强验证 Prompt。当确认没有具体目标诉求时也用作聊天伴侣。
- **主要引入包**：`openai.AsyncOpenAI`, `json`, `app.models.schemas.RouterResult`, `config`
- **包含的函数**：
  - `analyze_intent(user_input)`: 向大模型下达高严苛度的 System Prompt（要求提取具体工具或规则、投诉等语义）。结果强制打平为 Pydantic `RouterResult` 实体。
  - `generate_chitchat(user_input)`: 专精于闲聊意向处理，赋予大模型同理情感去拟人对话。

### 2. `app/services/rag_service.py`
- **主要作用**：公司化文档大杀器搜索库（Kb_qa）。专门应对“怎么报销”，“什么规定”等知识查找意图。本地整合 `ChromaDB` 持久化数据库查询和 `HuggingFace` 开源自部署轻量向量化模型（降本提效）。
- **主要引入包**：`os`, `chromadb`, `llama_index` 各生态扩展组件, `HuggingFaceEmbedding`
- **包含的函数**：
  - `build_or_load_index()`: 将 `data/uploads` 下存放的企业专属文档利用所选 HuggingFace 微型模型将其碎化、转变成数学向量库 `ChromaVectorStore` 对象返回。
  - `query_knowledge(question)`: 暴漏给 `chat.py` 内部调用，启动一个查询引擎直接将知识库比对文本连同查询诉求整合成回答内容反馈给上层。

### 3. `app/services/tool_service.py`
- **主要作用**：多步骤外部功能执行体 (ReAct Agent 操作间)。负责处理用户的查天气、查订单动作。按照通用工业级“Function Calling Format“协议把工具描述丢给大模型让其抽取代入参数并二次回复。
- **主要引入包**：`json`, `openai.OpenAI`
- **包含的函数**：
  - `get_weather(city)`: （私有）模拟��接气象开放接口行为的功能件。
  - `get_order_status(order_id)`: （私有）模拟连接物流接口查件能力的功能件。
  - `execute_tool_call(user_input)`: 该模块的主暴露接口。先组装带有多功能说明书 (`tools_schema`) 的消息发送大模型 -> 截取大模型反馈认为应执行的工具包名和参数 `tool_calls` -> 本地路由并调用具体业务函数取得结果 -> 将客观数据原形带入再次由大模型进行二次“通人言化”的口吻改造包装发回。

## 阶段四：UI 呈现层 (客户端)

### `frontend/streamlit_app.py`
- **主要作用**：提供一个快速可验证的客户端体验页面与开发者调试面板，独立运作运行（它本身并非 FastAPI 一部分而是平级的 HTTP 终端请求方）。
- **主要引入包**：`streamlit`, `requests`, `uuid`
- **页面动作逻辑**：
  - 启动后使用自带 Session 缓存避免重刷新引起状态清空。
  - 用户按下发送框后，程序包装 REST Payload 交给后端 `API_URL`（8000端口）。
  - 等待过程中渲染思考 Spinner。获取响应后向用户气泡区打印答复，同时展开隐藏的日志侧边栏面板暴漏请求中带有的所有内部状态（方便调试）。
