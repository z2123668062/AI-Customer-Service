# V3.0 第二期纯干货日记：异常处理体系与全链路可追溯

## 核心目标

填坑阶段的任务 1.3（异常处理修复）。三个目标：**出了错能定位到模块、出了错能追溯到请求、响应体中带语义化错误码。** 不改业务逻辑，不动端点接口，重构异常处理的基础设施。

三个文件新建（`exceptions.py`、`error_handlers.py`），两个文件改造（`main.py`、`chat.py`），一个线上 bug 修复（`redis-py` 锁的兼容性问题）。

---

## 核心变革一：业务异常基类

### 之前的问题

所有异常处理写在 `main.py` 里，三个全局异常处理器堆在一起。每个处理器返回的响应体中，`session_id` 写死为 `"unknown"`，没有 `trace_id`，没有 `error_code`，没有 `module`——前端收到错误不知道是什么类型的错，后端拿到日志不知道是哪个模块抛的。

抛出异常的方式完全是裸的 `raise HTTPException(status_code=429, detail="...")`，只能传状态码和字符串，没有"业务语义"。

### 修复方式

**新建 `app/core/exceptions.py`**，定义一个 `AppException` 基类：

```python
class AppException(Exception):
    def __init__(self, error_code: str, message: str, http_status: int = 500, module: str = "unknown"):
        self.error_code = error_code
        self.message = message
        self.http_status = http_status
        self.module = module
```

四个字段各有定位：
- `error_code`：机器读的，前端可以根据它做差异化处理（如 `RATE_LIMIT` 弹 Toast、`AUTH_FAILED` 跳登录页）
- `message`：人读的，直接展示给用户或运维
- `http_status`：HTTP 状态码，决定响应头的状态
- `module`：谁抛的，精确到模块名（如 `"rag_service"`、`"ratelimit_service"`）

设计上的取舍：没有做复杂的异常继承树（像 Java 那样 `RateLimitException extends AppException`）。因为当前项目只有 14 个文件，异常类型一只手数得过来，一个基类 + 四个字段足够覆盖。以后需要特殊处理某个异常类型时，再拆子类不迟。

## 核心变革二：异常处理器独立

### 之前的问题

三个异常处理器直接写在 `main.py` 的文件底部，紧挨着 `app.include_router` 的调用。随着项目变大，`main.py` 会变成"垃圾桶"——所有人都往里塞东西，冲突高发且难以维护。

### 修复方式

**新建 `app/core/error_handlers.py`**，把三个异常处理器从 `main.py` 搬出来，并做了三项核心升级：

**升级一：`session_id` 和 `trace_id` 从 `request.state` 读取，不再写死。**

FastAPI 的每个请求对象自带 `request.state` 属性——一个空的字典式对象，可以在请求的生命周期内往里面存任何数据，请求结束后自动销毁。`trace_id` 在中间件阶段写入，`session_id` 在 chat 端点解析出 JSON body 后写入。异常处理器在任何阶段触发，都能通过 `request.state` 读到——如果取不到就 fallback 到 `"unknown"`。

这里避开了 `contextvars`（Python 标准库的上下文变量方案），直接用 FastAPI 内置的 `request.state`。原因很简单：`request.state` 是框架原生支持的，生命周期和请求绑定，不需要手动清理。`contextvars` 更适合"跨框架、跨调用栈"的场景（比如代码同时被 FastAPI 和 Celery 调用），当前项目不需要。

**升级二：响应体增加 `error_code` 和 `module` 两个字段。**

`error_code` 按状态码映射：`422→VALIDATION_ERROR`、`429→RATE_LIMIT`、`401→AUTH_FAILED`、`403→FORBIDDEN`、`500→INTERNAL_ERROR`。

`module` 通过 URL 路径反推：`/api/v1/chat`→`chat_endpoint`、`/api/v1/auth`→`auth_endpoint`。

对于 `AppException`，`module` 直接从异常的字段读取——谁抛的谁填自己的模块名，能精确到 `"rag_service"` 而不是 `"chat_endpoint"`。

**升级三：`http_exception_handler` 同时处理两类异常。**

函数签名是 `async def http_exception_handler(request, exc)`——先判断 `exc` 是不是 `AppException`，如果是就走 `AppException` 的分支（用 `exc.module`、`exc.error_code`）；如果不是则按普通 `HTTPException` 处理（从 URL 反推 module、从状态码映射 error_code）。

**注册方式**：`error_handlers.py` 底部提供一个 `register_exception_handlers(app)` 函数，在 `main.py` 里调一行完成注册。不依赖装饰器，不依赖全局变量，`main.py` 只需要知道"有个注册函数"而不需要知道内部注册了几个 handler。

## 核心变革三：全局 `trace_id` 中间件

### 之前的问题

`trace_id` 只在 `chat.py` 里生成，只在那一个接口里使用。其他接口（auth、kb、sessions）完全不生成。异常处理器里也无法拿到。

### 修复方式

**`main.py` 新增 `trace_id_middleware`：**

```python
@app.middleware("http")
async def trace_id_middleware(request, call_next):
    import uuid
    request.state.trace_id = uuid.uuid4().hex
    response = await call_next(request)
    return response
```

这是一个全局中间件——所有 HTTP 请求进来，在进入任何端点之前，先经过这里。`trace_id` 在此时写入 `request.state`，后续不管走到哪个端点、哪个服务层、哪个异常处理器，都能通过 `request.state.trace_id` 拿到。

`uuid.uuid4()` 的生成开销大约是 0.3-0.5 微秒（百万分之零点几秒），对任何请求量级都不构成性能问题。

**`chat.py` 的改动：**
- `chat_endpoint` 不再自己生成 `trace_id`，改为从 `fastapi_request.state.trace_id` 读取（如果中间件没生成就兜底自己生成）
- 解析出 `request.session_id` 后写入 `fastapi_request.state.session_id`，打通异常处理器拿到 session_id 的链路

## 异常全链路追溯流程

```
请求进来 → trace_id_middleware
               ↓
          写入 state.trace_id
               ↓
          进入端点（chat/auth/kb/sessions 等）
               ↓
          chat_endpoint: state.session_id = request.session_id
               ↓
          业务逻辑执行...
               ↓
          ┌─ 正常 → 返回响应
          │
          └─ 抛异常 → FastAPI 匹配异常类型 → 对应的 handler
                     ↓
               从 state 读 trace_id + session_id
                     ↓
               组装 error_code + module
                     ↓
               JSONResponse 返回给前端
```

三种异常场景的响应示例：

参数校验失败（422）：
```json
{"session_id": "unknown", "trace_id": "a1b2c3", "error_code": "VALIDATION_ERROR", "module": "chat_endpoint"}
```
session_id 为 "unknown" 是合理的——请求连最基本的参数都不全，还没走到能解析 session_id 的代码。

限流触发（429）：
```json
{"session_id": "user_xxx", "trace_id": "a1b2c3", "error_code": "RATE_LIMIT", "module": "chat_endpoint"}
```
session_id 能从 state 拿到了，因为端点已经开始执行了。

业务异常（503）：
```json
{"session_id": "user_xxx", "trace_id": "a1b2c3", "error_code": "RAG_TIMEOUT", "module": "rag_service"}
```
module 精确到服务层，因为 `AppException` 抛异常时自己填了 `module="rag_service"`。

## 踩坑实录

### uuid.uuid4() 性能误判

分析异常链路时，最开始的担忧是"每个请求都调一次 uuid4，会不会成为瓶颈"——实际上 uuid4 的开销是微秒级的，不构成性能问题。这个担忧来源于对"系统调用"的直觉，但 uuid4 是纯算法生成，不需要调用任何系统接口（不像 `/dev/urandom`），所以开销极低。如果这个级别的开销都要担心，那程序里任何一个函数调用都不应该写。

### request.state 的可写时机

FastAPI 的 `request.state` 在中间件阶段可以写入，在端点函数中也可以写入。但要注意的是：写入必须发生在读取之前。如果异常在 `trace_id_middleware` 之前就触发了（目前不可能，因为中间件是入口第一关），那 `request.state` 上就没有 `trace_id`。当前的设计保证了 `trace_id` 是所有异常处理器的"安全字段"——中间件在第一个就执行，100% 写入后才进入后续任何代码。

### JSONResponse 的 content 参数

练习时犯了一个典型错误：把 `JSONResponse(trace_id=xxx)` 写成了关键字参数。`JSONResponse` 只接受 `content` 作为数据参数，其他都是 HTTP 层面的参数（`status_code`、`headers` 等）。正确的写法是 `JSONResponse(content={"trace_id": xxx})`。这个错误的根因是把 JSONResponse 当成了一个"序列化器"而不是"HTTP 响应构造器"。

## 架构反思

### 1. 异常处理器的"注册"与"定义"分离

把异常处理器定义和注册行为拆开，是本次重构最值得重复的模式。`error_handlers.py` 定义处理器逻辑，`main.py` 只需调一行 `register_exception_handlers(app)`。结果是：

- `main.py` 不需要知道内部注册了几个 handler
- 新增异常类型时只需改 `error_handlers.py`，`main.py` 零改动
- 测试可以直接 import 单个 handler 函数测试，不需要启动整个 app

### 2. request.state 与 contextvars 的选择

FastAPI 的 `request.state` 和 Python 的 `contextvars` 都能做"请求级别的上下文"。区别在于：

- `request.state` 强依赖 FastAPI 框架，好处是框架自动管理生命周期，缺点是脱离了 FastAPI 的代码（如数据库驱动、消息队列消费者）拿不到它
- `contextvars` 不依赖框架，但需要在入口处 set、退出时 reset，少一步就内存泄漏

当前项目全在 FastAPI 生态内，`request.state` 是最合适的选择。如果以后引入 Celery 或异步消息队列，再改用 `contextvars` 也不迟。

### 3. 异常处理器的 async 考虑

异常处理器中不应该做网络 IO 或耗时操作。如果以后要在 500 错误时推送企业微信告警，正确的做法是把告警任务丢到后台执行（`asyncio.create_task` 或 `BackgroundTasks`），让异常处理器只负责组装 JSON 返回，不阻塞当前请求的响应。
